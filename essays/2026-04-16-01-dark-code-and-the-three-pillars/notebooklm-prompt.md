Focus on THREE main points:

1. **The Amazon disaster that revealed the problem.** In December 2025, Amazon's AI coding tool Kira autonomously deleted and recreated an AWS Cost Explorer environment, causing a 13-hour outage. Three months later, Amazon's website went dark — missing prices, broken checkout, failed deliveries. Internal documents cited "Gen-AI assisted changes." Between these incidents, Amazon fired 16,000 engineers. The code that broke everything was never understood by anyone — AI-generated, passed automated checks, shipped. Nate Jones named this "dark code": production software where the comprehension step never happened. This is not an Amazon problem. Every company shipping AI-generated code faces the same dilemma: slow down and lose to competitors, or accept code nobody reads and accept unquantified liability.

2. **The three pillars framework: spec + tests + code.** The essay's core insight: stop treating code as the thing that matters most and start treating it as the thing that matters least. Every software system rests on three pillars — Specification (what it should do), Tests (verification it does it), and Code (the implementation). The revolutionary claim: any two pillars should be strong enough to reconstruct the third. Spec + Tests can regenerate code. Spec + Code can derive tests. Tests + Code can extract the spec. StrongDM runs a "software factory" — $1,000/day in tokens, 16,000 lines of Rust daily, with two rules: "code must not be written by humans" and "code must not be reviewed by humans." RecursiveAI rebuilt a 20,000-line system in three days from specifications alone. Code becomes disposable. Specifications become the source of truth.

3. **Five assurance levels and the trust problem.** The framework maps to five workflow levels: Level 0 (vibe coding — no verification), Level 1 (assisted development — traditional review), Level 2 (spec-first with automated testing), Level 3 (dark factory with behavioral holdout scenarios invisible to coding agents), and Level 4 (formal mathematical verification / vericoding). The trust problem is real: when AI writes code AND tests, they share blind spots — Stanford calls this "circular validation." The solution is independence between pillars. External holdout scenarios break the circle. The human role moves up the stack: you stop reviewing code and start reviewing specifications, verification strategies, and independence guarantees. New roles emerge: specification architect and verification strategist.
