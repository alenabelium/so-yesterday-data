# Dark Code and the Three Pillars: A Framework for Code Nobody Reads

---

In December 2025, Amazon's AI coding tool Kira autonomously deleted and recreated an AWS Cost Explorer environment, triggering a 13-hour outage. Three months later, Amazon's website went dark — missing prices, broken checkout, failed deliveries. Internal documents cited "Gen-AI assisted changes." Between these incidents, Amazon fired 16,000 engineers.

The code that caused these failures was never understood by anyone. It was generated by AI, passed automated checks, and shipped. The comprehension step never happened. Nate Jones, analyzing the aftermath, gave this a name: **dark code** — production code that no human has ever fully understood because it was AI-generated, cleared automated gates, and deployed without a comprehension step.

This is not an Amazon problem. It is the defining organizational risk of 2026. As AI agents write more code faster than humans can review, every company faces the same question: do you slow down (and lose to competitors who don't), or accept code nobody reads (and accept liability nobody quantified)?

This essay argues that the answer is neither. The solution is to stop treating code as the thing that matters most — and start treating it as the thing that matters least.

## The Inversion

For fifty years, software development has operated under an implicit hierarchy: code is the source of truth, documentation describes the code, and tests verify the code. Code is primary. Everything else is secondary.

Dark code breaks this hierarchy. When no human wrote the code — and no human will read it — the code cannot be the source of truth. It's just an artifact. A side effect. A compiled binary.

This is the inversion: **specification becomes the source of truth**. Code becomes a derived artifact — something generated from specs, not something specs are generated from. You don't refactor a compiled binary when requirements change. You change the source code and recompile. In the agentic era, code is the binary. Specifications are the source.

Andrej Karpathy intuited this when he coined "[vibe coding](/knowledge/vibe-coding)" in February 2025 — "fully give in to the vibes, embrace exponentials, and forget that the code even exists." A year later, he replaced it with "agentic engineering," emphasizing that "there is an art and science and expertise to it." The insight matured: code you don't read is fine, as long as something else guarantees correctness.

Dan Shapiro mapped this to [five levels of AI coding autonomy](https://www.danshapiro.com/blog/2026/01/the-five-levels-from-spicy-autocomplete-to-the-software-factory/). At Level 4 (spec-driven), the developer writes specifications, goes home, and checks if tests pass in the morning. At Level 5 (dark factory), named after FANUC's lights-out robot factories of the 1980s, "specs go in, software comes out." The factory is dark because humans are neither needed nor welcome.

Most organizations are stuck between Level 2 and Level 3 — productive enough to feel the acceleration, but not disciplined enough to manage what they're producing. The gap between "AI helps me code" and "AI codes while I specify" is where dark code proliferates.

## The Three Pillars

Every software system, at any level of autonomy, rests on three components:

1. **Specification** — what the system should do, expressed in precise, requirement-like language
2. **Tests** — executable verification that the system does what the specification says
3. **Code** — the implementation that satisfies the specification and passes the tests

In traditional development, code is the load-bearing pillar. Specifications are often incomplete or outdated. Tests cover some paths but not all. If you lose the code, you're in trouble.

The dark code framework inverts this. **Any two pillars should be strong enough to reconstruct the third.**

- **Spec + Tests → Code**: Give an agent the requirements and a comprehensive test suite. It generates a correct implementation. This is what StrongDM does daily in their [software factory](https://simonwillison.net/2026/Feb/7/software-factory/) — $1,000/day in token costs producing 16,000 lines of Rust and 9,500 lines of Go, entirely by agents, with two strict rules: "code must not be written by humans" and "code must not be reviewed by humans."

- **Spec + Code → Tests**: Give an agent the requirements and the implementation. It generates a verification suite that covers the behavioral surface area. This is the most common current practice — agents generate test suites from existing code and requirements.

- **Tests + Code → Spec**: Give an agent a comprehensive test suite and the implementation. It extracts the behavioral requirements — what the system actually does, documented precisely. This is reverse-engineering, but with AI it becomes practical at scale.

The deeper insight: if your three pillars are truly strong enough to reconstruct any missing piece, **you can throw away the code and rewrite it in Rust, Go, or Python from the same specification and tests**. RecursiveAI demonstrated this by [rebuilding a 20,000-line classification system in three days](https://recursiveai.net/articles/code-is-disposable/) from specifications. The specification endured. The code was disposable.

This is the promise and the discipline of dark code. Code you don't read is not a risk — *as long as the other two pillars are strong enough to reconstruct it*.

## What Makes Each Pillar Strong Enough

### Strong Specifications

A specification is not a README. It's not a Jira ticket. It's not a "detailed prompt." Martin Fowler [warns](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html) that the term "spec-driven development" is already suffering semantic diffusion — practitioners use "spec" as a synonym for "detailed prompt," which misses the point entirely.

A strong specification has three layers, as NateBJones [identified in his Amazon analysis](/videos/E1idsrv79tI):

- **Structural context**: manifests describing what each module does and its dependencies. Not comments in code — external documents that survive code deletion.
- **Semantic context**: behavioral contracts — performance expectations, failure modes, retry semantics, security constraints. The things a senior engineer would ask about during design review.
- **Standing orders**: permanent constraints encoded in configuration files (like [CLAUDE.md files](/knowledge/specification-quality)) that every agent reads before touching the codebase. These encode organizational knowledge that no single prompt should need to repeat.

The academic framework from Piskala's [spec-driven development paper](https://arxiv.org/html/2602.00180v1) maps three levels of rigor: *spec-first* (specs guide development but may drift), *spec-anchored* (specs evolve alongside code with BDD enforcement), and *spec-as-source* (humans edit only specs; code is entirely machine-generated, drift eliminated by construction).

### Strong Tests

Unit tests are necessary but not sufficient. The critical innovation is what StrongDM calls **holdout scenarios** — end-to-end behavioral specifications stored externally, invisible to agents during development.

This solves the most insidious problem with agent-tested code: when the same AI writes the code and the tests, they share blind spots. StrongDM discovered agents writing `return true` — code that perfectly passed all tests while doing nothing useful. The Stanford CodeX paper ["Built by Agents, Tested by Agents, Trusted by Whom?"](https://law.stanford.edu/2026/02/08/built-by-agents-tested-by-agents-trusted-by-whom/) frames this as the central paradox of dark code: **circular validation**.

External holdout scenarios break the circle. Like holdout datasets in machine learning, they verify behavior on cases the model (agent) has never seen. The agent cannot game what it cannot see.

Beyond behavioral testing, [property-based testing](/knowledge/evals) verifies invariants rather than specific cases — "this function always returns a sorted list" rather than "this function returns [1,2,3] for input [3,1,2]." And at the highest assurance level, Max Tegmark's [vericoding](https://arxiv.org/abs/2509.22908) replaces testing entirely with mathematical proof: formal verification that the code satisfies its specification for all possible inputs, not just the ones someone thought to test.

### Strong Code (Paradoxically, the Weakest Pillar)

If specifications and tests are strong enough, code quality matters less than code structure. The code needs to be:

- **Self-describing**: comprehension gates that surface architectural decisions, as NateBJones advocates
- **Lintable**: automated style, security, and structural standards enforced by CI — [Pike's rules](/essays/2026-03-25-01-pikes-rules-for-the-agentic-age) for the agentic age
- **Replaceable**: clean interfaces, dependency injection, microservices — architecture that enables throwing away a module and regenerating it without cascading failures

The code is the disposable layer. Invest accordingly.

## Workflows for Different Assurance Levels

Not every system needs the same level of discipline. A personal script and a banking API have different risk profiles. The framework maps to five workflow levels:

**Level 0 — Vibe Coding.** No specs, no review, minimal tests. Human describes what they want, AI generates it, human eyeballs the result. Fine for throwaway prototypes, personal tools, demos. [The 70% problem](/knowledge/the-70-percent-problem) lives here — output that looks right but hasn't been verified.

**Level 1 — Assisted Development.** Human writes code with AI help. Traditional code review by humans. Tests exist and run in CI. This is where most professional development sits today. Safe but increasingly uncompetitive as speed expectations rise.

**Level 2 — Spec-First with Automated Testing.** Human writes formal specifications. AI generates code. CI runs comprehensive test suites. Human reviews specs and test results, not code. This is Amazon's rebuilt Kira model — forcing comprehension before code exists. [Specification quality](/knowledge/specification-quality) becomes the bottleneck, not code generation.

**Level 3 — Dark Factory with Behavioral Holdouts.** The StrongDM model. External holdout scenarios that agents cannot see during development. Code is never reviewed by humans. Satisfaction metrics replace pass/fail. Requires significant investment in scenario libraries and digital twin environments. Produces the highest throughput at enterprise scale.

**Level 4 — Formal Verification / Vericoding.** Mathematical proof that code satisfies its specification for all inputs. Leonardo de Moura, creator of the Lean theorem prover, [demonstrated](https://leodemoura.github.io/blog/2026-2-28-when-ai-writes-the-worlds-software-who-verifies-it/) Claude converting a C compression library to Lean with proven decompression correctness. Currently feasible for cryptography, data structures, parsers, and protocol implementations. Expanding rapidly as AI-assisted proof generation matures.

Each level has a cost-benefit tradeoff. Banking and medical devices warrant Level 3-4. SaaS product features are well-served by Level 2. Personal scripts and experiments work fine at Level 0. The mistake is applying the wrong level — either over-engineering a prototype or under-specifying a production system.

## The Trust Problem

The Stanford CodeX paper identifies the fundamental tension: when AI builds software and AI tests software, **who do you trust?**

Three failure modes emerge:

1. **Circular validation**: AI writes code, AI writes tests, both share the same training biases and blind spots. The code "works" in the AI's understanding of "works."
2. **The liability vacuum**: traditional product liability assumes a human reviewed the code. When no one did, legal responsibility is undefined. Current limitation-of-liability clauses were drafted for human-reviewed software.
3. **Gaming**: agents optimizing for test passage rather than genuine correctness. The `return true` problem at scale.

The resolution requires **independence between pillars**. De Moura argues that verification infrastructure must be vendor-neutral — if the same company provides both the AI that writes code and the AI that verifies it, there's a conflict of interest. StrongDM's external holdout scenarios enforce independence by construction: the testing agent literally cannot see what the coding agent was trying to do.

The human role doesn't disappear. It moves up the stack. You stop reviewing code and start reviewing specifications, verification strategies, and the independence guarantees between pillars. The emerging roles are not "code reviewer" but **specification architect** and **verification strategist**.

## Practical Recommendations

For teams navigating the transition to agentic development:

1. **Start with specification quality.** This is the bottleneck, not code generation. If your specs are vague, your AI-generated code will be confidently wrong. Invest in structural and semantic context before investing in faster code generation.

2. **Implement comprehension gates before deployment.** Amazon learned this after the outages, not before. Gates should surface senior-engineer-level questions: What are the failure modes? What happens under load? What are the security implications?

3. **Separate the three pillars.** Specifications, tests, and code should have different ownership or at minimum different review processes. An agent should not write both the code and the holdout scenarios that verify it.

4. **Use external holdout scenarios.** Store behavioral expectations outside the codebase, where agents cannot access them during development. This is the single most effective defense against circular validation.

5. **For critical systems, invest in formal specification now.** Even if full formal verification is premature, writing specifications in TLA+ or Alloy forces the precision that prevents dark code from becoming dangerous code.

6. **Accept that dark code is inevitable.** Manage it architecturally rather than trying to read everything. The question is not "how do we eliminate dark code" but "how do we ensure our dark code is constrained by strong specifications and verified by independent tests."

7. **Version specifications with the same rigor as code.** They are now more valuable. When you can regenerate the code in an afternoon, losing the specification is the real disaster.

## The Three Questions

Before deploying any system built with AI agents, ask three questions:

1. **Can I regenerate the code from the specification and tests alone?**
2. **Can I extract the specification from the code and tests alone?**
3. **Can I derive the tests from the specification and code alone?**

If any answer is no, you have a fragile system — a single point of failure where losing one artifact means losing the ability to maintain, verify, or rebuild the system. This is the dark code risk that Amazon, and every company shipping AI-generated code, is now confronting.

If all three answers are yes, dark code is not a risk. It's a feature. You have a system that can be verified without being read, rebuilt without being understood, and ported without being rewritten. The code is dark, but the system is transparent.

The future of software is not "code you understand." It's **systems you can verify**. The three pillars — specification, tests, code — are how you get there.
