Disposable software is the concept of building applications quickly with AI tools for a specific, often temporary purpose — with the explicit intention that the software will be thrown away rather than maintained long-term. This inverts the traditional software engineering mindset where every piece of code is built to last.

The economics of software have changed: when AI can generate a working application in minutes, it can be cheaper to regenerate software from specifications than to maintain it. However, 90% of people get the concept wrong by applying it where maintainability matters. Disposable software works for one-off analyses, prototypes, internal tools, and personal utilities — not for production systems that users depend on. [Vibe coding](/knowledge/vibe-coding) is the primary technique used to create disposable software.

## Key Aspects

- **Regeneration over maintenance** — when the spec is good, regenerating is cheaper than debugging
- **Appropriate contexts** — prototypes, data analysis scripts, personal tools, demos — not production systems
- **Hidden costs** — the trend has hidden costs around data integrity, security, and accumulated technical expectations
- **Specification as the artifact** — in a disposable software world, the specification becomes more valuable than the code

The [Builder](/personas/03-builder) persona must navigate when disposable software is appropriate vs. when production discipline is needed. The Builder's 'corner questions' practice — proactively asking about error handling, security, and scale — is the filter that separates acceptable disposable software from reckless shortcuts.
