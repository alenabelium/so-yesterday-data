Dark code is production software that was never understood by any human at any point — it was generated by AI, passed automated checks, and shipped. The comprehension step didn't happen. Unlike traditional technical debt, where code was once understood but knowledge decayed, dark code's comprehension gap exists from the moment of creation.

The term gained prominence in April 2026 after Amazon's AI coding tool Kira triggered major outages following the layoff of 16,000 engineers. Dark code multiplies for two reinforcing reasons: AI-generated code is structurally harder to understand because you did not write it, and organizational pressure to move fast means comprehension decouples from authorship. Layoffs accelerate the problem by burdening fewer engineers with more unreviewed code.

The solution framework rests on three pillars — specification, tests, and code — where any two can reconstruct the third. If specifications and tests are strong enough, code becomes a disposable artifact: you can throw it away and regenerate it in any language. This reframes dark code from a defect to be eliminated into a condition to be managed through architectural discipline, with different assurance levels appropriate for different risk profiles.

## Key Aspects

- **The three pillars** — Specification (what the system should do), Tests (verification it does it), Code (the disposable implementation). Any two should be able to reconstruct the third. ([Specification Quality](/knowledge/specification-quality))
- **The inversion** — Code is no longer the source of truth. Specifications are. Code is a derived artifact, like a compiled binary — you don't refactor the binary, you change the source and recompile. ([Disposable Software](/knowledge/disposable-software))
- **Assurance levels** — From Level 0 ([vibe coding](/knowledge/vibe-coding) with no verification) through Level 4 (formal mathematical proof / vericoding). The right level depends on the risk profile of the system.
- **Circular validation risk** — When the same AI writes code and tests, they share blind spots. External holdout scenarios (behavioral tests invisible to the coding agent) break this circle.
